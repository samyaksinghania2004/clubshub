\section{Introduction}

\productname{} is a web application for IIT Kanpur clubs, events, open discussion rooms, notifications, and direct messages. The system is accessible from both desktop and mobile browsers, and it also supports installable Progressive Web App behavior on supported devices.

This manual focuses on actual user workflows available in the current implementation:

\begin{itemize}[leftmargin=1.5em]
  \item account creation and login
  \item event discovery and registration
  \item club membership and club channel usage
  \item anonymous open-room participation
  \item direct messages
  \item notifications, theme toggle, and mobile usage
  \item coordinator and administrator actions that affect normal users
\end{itemize}

\subsection{Supported roles}

The system currently uses the following roles:

\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Student}: normal end user
  \item \textbf{Club member}: student with active membership in a club
  \item \textbf{Club secretary}: club-level manager with event and channel responsibilities
  \item \textbf{Club coordinator}: club-level manager with stronger membership and channel control
  \item \textbf{Institute admin / system admin}: elevated administrative roles
\end{itemize}

The interface shown to a user depends on the account role and the user's status inside each club or room.

\section{Getting Started}

\subsection{Opening the application}

Open the ClubsHub URL in a browser. After the page loads, the shared application shell appears:

\begin{itemize}[leftmargin=1.5em]
  \item left sidebar for main navigation
  \item top search bar for clubs, events, and open rooms
  \item notifications button
  \item theme toggle
  \item user/profile shortcut
\end{itemize}

\subsection{Main navigation}

The current navigation options are:

\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Events}: browse upcoming events
  \item \textbf{Clubs}: browse clubs and open club pages
  \item \textbf{Open rooms}: browse discussion rooms
  \item \textbf{My activity}: profile-style summary of memberships, registrations, and room handles
  \item \textbf{Inbox}: direct messages
\end{itemize}

On mobile devices, the left sidebar opens as a drawer from the menu button.

\section{Account Access and Authentication}

\subsection{Creating an account}

New users must create an account using an \texttt{@iitk.ac.in} email address.

\begin{enumerate}[leftmargin=1.8em]
  \item Open the sign-up page.
  \item Enter username, IITK email address, and password.
  \item Submit the form.
  \item Open the verification email.
  \item Click the verification link to activate the account.
\end{enumerate}

If the user does not verify the account in time, the system provides a resend-verification flow. If the verification email was not expected, the user can report the signup from the email itself.

\subsection{Password login}

Users can sign in with either:

\begin{itemize}[leftmargin=1.5em]
  \item username and password, or
  \item IITK email and password
\end{itemize}

Only verified accounts can complete password login.

\subsection{OTP login}

Verified users can also log in using a one-time code sent by email.

\begin{enumerate}[leftmargin=1.8em]
  \item Open the login page.
  \item Click \textbf{Use OTP instead}.
  \item Enter the IITK email address.
  \item Submit the request.
  \item Open the email and note the 6-digit code.
  \item Enter the code on the OTP verification page.
\end{enumerate}

\subsection{Forgot password}

The password reset flow is available from the login page.

\begin{enumerate}[leftmargin=1.8em]
  \item Click \textbf{Forgot password?}
  \item Enter the email address associated with the account.
  \item Open the password reset email.
  \item Follow the link and set a new password.
\end{enumerate}

\subsection{Verification pending state}

Immediately after account creation, the system sends the user to a pending-verification page. This page reminds the user to check email and provides a resend path.

\section{Dashboard Layout and Common Controls}

\subsection{Top bar}

The top bar contains the most frequently used controls:

\begin{itemize}[leftmargin=1.5em]

  \item theme toggle

On smaller screens, the top bar becomes more compact, but the same controls remain available.

\subsection{Theme toggle}

Users can switch between light and dark appearance at any time. The selected theme is stored in the browser, so it remains active on the same device until changed again.

\subsection{Notifications}

The notifications page collects system updates such as:

\begin{itemize}[leftmargin=1.5em]
  \item event-related updates
  \item room invitations or outcomes
  \item moderation outcomes
  \item announcement-related updates
\end{itemize}

Unread notifications are indicated by a badge on the notifications button.

\begin{itemize}[leftmargin=1.5em]
  \item browse published events
  \item refine the feed using filters
  \item view personal registration status
  \item open an event detail page
  \item jump to the event discussion channel when available

The event detail page shows:

\begin{itemize}[leftmargin=1.5em]
  \item event title
  \item organizing club
  \item about/description
  \item venue
  \item schedule
  \item registration summary
  \item announcements
  \item personal registration status

Announcements are intentionally compact in the main page. Users click an announcement title to open the full announcement body in a dialog instead of expanding long text directly in the page.

\subsection{Registering for an event}

To register:

\begin{enumerate}[leftmargin=1.8em]
  \item open the event detail page
  \item click \textbf{Register}
  \item wait for the page to refresh
  \item confirm the updated registration status
\end{enumerate}

If the event reaches capacity, the system may place the user on a waitlist depending on the event state. If a confirmed participant cancels, the system may promote users from the waitlist automatically.

\subsection{Canceling a registration}

Users can cancel their own registration from the event detail page. The page then updates the personal status block accordingly.

\subsection{Attendance and analytics}

Attendance management and analytics are role-restricted features. Normal students mainly interact with:

\begin{itemize}[leftmargin=1.5em]
  \item registration state
  \item cancellation
  \item announcements
  \item discussion link
\end{itemize}

\section{Clubs and Club Channels}

\subsection{Browsing clubs}

The \textbf{Clubs} page shows available clubs and supports search by name or category. Users can open a club page from the list.

\subsection{Joining and leaving a club}

From a club page, users can join or leave depending on their current membership state.

Important current behavior:

\begin{itemize}[leftmargin=1.5em]
  \item normal members can leave a club voluntarily
  \item if a coordinator removes a user from a club, the user cannot rejoin independently
  \item removed members must be restored by a coordinator or administrator
\end{itemize}

\subsection{Club channel workspace}

The club page uses a chat-style workspace:

\begin{itemize}[leftmargin=1.5em]
  \item the main area shows the active channel
  \item newer messages are presented at the bottom of the conversation, and the workspace opens at the latest messages
  \item older messages are reached by scrolling upward inside the chat area
  \item the right panel shows channels and member information
\end{itemize}

On mobile, the right panel becomes a collapsible \textbf{Info} drawer so the user does not need to scroll below the message composer.

\begin{itemize}[leftmargin=1.5em]

\subsection{Editing and deleting messages}

Users can edit their own room messages for a short time after posting. Users can also delete their own room messages, while room managers and administrators may have stronger moderation controls.

The room creator can manage certain room-specific actions. In the current implementation, the room creator can use a combined \textbf{reveal and expel} action for a participant when appropriate. This action is room-specific and is not available to all participants.

\subsection{Archived rooms}

Archived rooms are read-only. Users can still view earlier conversation history, but the message composer is no longer available for sending new messages.

\section{Direct Messages}

\subsection{Opening the inbox}

The \textbf{Inbox} page is the direct-message interface. It supports:

\begin{itemize}[leftmargin=1.5em]
  \item conversation list
  \item active thread workspace
  \item new chat search
  \item block or unblock actions for a thread
\end{itemize}

The inbox was redesigned to behave like a modern chat workspace:

\begin{itemize}[leftmargin=1.5em]
  \item the thread opens at the latest messages
  \item older messages are reached by scrolling upward inside the thread
  \item the conversation list stays accessible
  \item on mobile, the thread occupies most of the screen
\end{itemize}

\section{Administrator Panel}

\subsection{Purpose}

ClubsHub also includes the standard Django administrator panel at:

\begin{center}
  \texttt{/admin/}
\end{center}

This interface is intended for desktop use. It is useful for administrative maintenance, data inspection, and support actions that are broader than the normal user interface.

\subsection{What is available in the admin panel}

The current admin surface exposes the active operational models used by the project, including:

\begin{itemize}[leftmargin=1.5em]
  \item users and OTP login challenges
  \item clubs, memberships, events, registrations, announcements, channels, channel members, and channel messages
  \item rooms, room handles, invites, room messages, and reports
  \item notifications and audit logs
  \item direct-message threads, participants, messages, and block relationships
\end{itemize}

Stale or confusing legacy items are intentionally hidden. In particular:

\begin{itemize}[leftmargin=1.5em]
  \item the legacy \texttt{ClubFollow} model is not exposed in admin
  \item Django auth \texttt{Groups} are not used by the project permission model and are hidden from admin
\end{itemize}

\subsection{Recommended admin uses}

The admin panel is best used for:

\begin{itemize}[leftmargin=1.5em]
  \item inspecting account and membership records
  \item checking event registrations and announcements
  \item reviewing room reports, handles, invites, and archived state
  \item auditing notifications, direct messages, and moderation history
  \item data correction when a support issue cannot be resolved from the normal UI
\end{itemize}

Audit-style records are intended mainly for inspection rather than day-to-day editing.

\section{Mobile and Installable App Use}

\subsection{Responsive behavior}

ClubsHub supports both desktop and mobile use. In mobile mode:

\begin{itemize}[leftmargin=1.5em]
  \item the left navigation becomes a drawer
  \item chat surfaces prioritize the message stream
  \item club and room side information moves into collapsible \textbf{Info} drawers
  \item the message composer remains easy to reach while typing
\end{itemize}

\subsection{Installable app behavior}

Supported browsers may offer an \textbf{Install app} prompt. When installed, ClubsHub behaves like a home-screen web app rather than a native Android or iOS application.

\section{Troubleshooting}

\subsection{I did not receive a verification email}

\begin{itemize}[leftmargin=1.5em]
  \item confirm that the signup used an \texttt{@iitk.ac.in} address
  \item check spam or junk folders
  \item use the resend-verification page if the account is eligible
\end{itemize}

\subsection{I cannot log in even though I created an account}

Make sure the email verification link was completed successfully. Unverified accounts cannot complete normal password login.

\subsection{I was removed from a club}

Removed members cannot currently rejoin by themselves. A club coordinator or administrator must restore the membership.

\subsection{I cannot send a message in a room}

Possible reasons include:

\begin{itemize}[leftmargin=1.5em]
  \item the room is archived and read-only
  \item the user no longer has an active room handle
  \item the user was expelled from the room
\end{itemize}

\subsection{The mobile page still shows an older layout}

If the interface looks outdated after deployment:

\begin{itemize}[leftmargin=1.5em]
  \item refresh the page more than once
  \item clear the browser's site data
  \item if installed as a web app, close and reopen it after clearing cached data
\end{itemize}
